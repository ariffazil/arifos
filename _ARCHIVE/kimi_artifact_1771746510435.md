## 5. API Reference

### 5.1 Protocol Fundamentals

#### 5.1.1 Transport Protocols

##### 5.1.1.1 JSON-RPC 2.0 Specification Compliance

arifOS MCP implements **full JSON-RPC 2.0**  [(aerospike.com)](https://aerospike.com/blog/model-context-protocol/)   [(modelcontextprotocol.io)](https://modelcontextprotocol.io/specification/draft/server/tools) : request/response with `id` correlation, batch requests, error objects with structured `data`, and notification support. All messages are UTF-8 encoded with `Content-Type: application/json`.

##### 5.1.1.2 SSE Stream Format: Event Types, Reconnection Handling

SSE events follow standard format with arifOS-specific types:

| Event Type | Purpose | Payload |
|-----------|---------|---------|
| `init` | Session established | `{"session_id": "...", "floors": [...]}` |
| `progress` | Stage completion update | `{"stage": "agi_reason", "completion": 0.6}` |
| `verdict` | Final output ready | `{"status": "SEALED", "content": "...", "audit_id": "..."}` |
| `error` | Failure notification | `{"code": -32001, "message": "...", "data": {...}}` |
| `ping` | Connection keepalive | `{}` |

Reconnection: clients should implement exponential backoff with `Last-Event-ID` header for replay.

##### 5.1.1.3 HTTP Long-Polling Fallback

For environments blocking SSE, HTTP transport supports long-polling via `/messages/poll` with `?since=<message_id>` parameter.

#### 5.1.2 Core Endpoints

##### 5.1.2.1 `POST /api` — Primary JSON-RPC Endpoint

Accepts standard JSON-RPC requests, returns responses or errors. Supports batch requests (array of request objects).

##### 5.1.2.2 `GET /sse` — Server-Sent Events Stream

Establishes persistent event stream. Requires `Accept: text/event-stream` header. Query parameters: `session_id` to attach to existing session.

##### 5.1.2.3 `POST /messages` — Message Ingestion Endpoint

Alternative entry for clients without full JSON-RPC implementation. Accepts simplified message format, returns acknowledgment.

##### 5.1.2.4 `GET /health` — Service Health Verification

Returns comprehensive status including subsystem health. Used for load balancer probes and monitoring dashboards  [(arifOS Project)](https://arifos.arif-fazil.com/docs)   [(PyPI)](https://pypi.org/project/arifos/) .

##### 6.3.3.3 Resolution: Implement Client-Side Reconnection, Adjust Heartbeat Intervals

```javascript
// Robust SSE client with reconnection
function connectSSE(url, lastEventId = null) {
  const headers = lastEventId ? {'Last-Event-ID': lastEventId} : {};
  const es = new EventSource(url, {headers});
  
  es.addEventListener('error', (e) => {
    if (es.readyState === EventSource.CLOSED) {
      setTimeout(() => connectSSE(url, lastEventId), 1000 * Math.pow(2, retryCount++));
    }
  });
  
  es.addEventListener('message', (e) => {
    lastEventId = e.lastEventId;
    retryCount = 0;  // Reset on success
    handleMessage(e.data);
  });
  
  return es;
}
```

##### 6.5.2.2 Response Fields: `status`, `version`, `active_sessions`, `metabolic_health`

Example response  [(arifOS Project)](https://arifos.arif-fazil.com/docs)   [(PyPI)](https://pypi.org/project/arifos/) :
```json
{
  "status": "healthy",
  "service": "aaa-mcp-rest",
  "version": "2026.02.17-FORGE-VPS-SEAL",
  "postgres": "connected",
  "redis": "connected",
  "active_sessions": 47,
  "metabolic_health": {
    "floor_violation_rate": 0.023,
    "mean_latency_ms": 145,
    "cooling_efficiency": 0.97,
    "constitutional_fidelity": 0.94
  }
}
```

##### 6.5.2.3 Integration: Load Balancer Health Probes, Monitoring Dashboards

Kubernetes probe configuration:
```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  failureThreshold: 3
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5
```

Prometheus metrics scraping:
```yaml
annotations:
  prometheus.io/scrape: "true"
  prometheus.io/path: "/metrics/json"
  prometheus.io/port: "8080"
```
