# 001 — GEOX expansion path: 8 branches toward "entire Earth"

> Current GEOX is strongest in subsurface/resource Earth. To become
> "entire Earth operating intelligence," these 8 branches need to grow
> from the same kernel.

## Branches

```
GEOX-SUBSURFACE   seismic, wells, basins, resources   — STRONG (current 8.0)
GEOX-SURFACE      geomorphology, mapping, remote sensing — to build (4.0)
GEOX-WATER        hydrology, groundwater, rivers, flood — to build (3.0)
GEOX-CLIMATE      atmosphere, ocean, carbon, climate risk — to build (2.0)
GEOX-ECO          soils, ecology, land use, biodiversity — to build (2.5)
GEOX-HAZARD       earthquake, landslide, volcano, tsunami — to build (4.0)
GEOX-MINERALS     mining, critical minerals, geochemistry — to build (4.0)
GEOX-INFRA        terrain, geohazard, foundations, cities — to build (?)
```

## Each branch follows the same pattern

For every new branch:

```text
1. Domain ontology:   what kinds of things exist
2. Computational:      can we calculate, inspect, transform
3. Governed-action:   can an agent act on this, under what evidence
4. Risk class:        irreversibility, blast radius, audit trail
5. Integration:       connects to GEOX-SUBSURFACE, then to arifOS / WEALTH / WELL
```

## Pattern of growth

```text
Phase 1: SUBSURFACE     — done (current 8.0)
Phase 2: HAZARD + MINERALS — already in scope, deepen (4.0 → 7.0)
Phase 3: SURFACE + WATER — adjacent to subsurface, easier bridge (4.0 → 6.5)
Phase 4: CLIMATE + ECO  — farthest from current kernel, biggest lift (2.0 → 5.0)
```

## Anti-pattern (what NOT to do)

- Don't build GEOX-CLIMATE first — too far from existing kernel
- Don't add 50 new tools at once — design branch by branch
- Don't claim GEOX is "entire Earth" before SURFACE branch exists

## Causal model: why this expansion order

```
SUBSURFACE = current strength; HAZARD and MINERALS are adjacent
SURFACE + WATER = same data substrate (geospatial, remote sensing)
CLIMATE + ECO = new data substrate, new ontologies
INFRA = depends on SURFACE + HAZARD + INFRA orthogonality
```

The closer to current kernel, the easier. The farther, the more new organ-level work.
